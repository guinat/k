# Risk Register

This register tracks the risks identified for the storage capacity management and deviation-detection system across the heterogeneous multi-OS fleet (AIX, all Linux distributions, and Windows Server, all first-class and delivered together). It covers data-quality, schema-integrity, collection, scalability, security, compliance, forecasting, standards, AI/agentic, and delivery risks. The project is in the scoping / specification phase, so every risk currently reflects a design-time or anticipated operational concern rather than an incident in a running system.

The severity column combines likelihood and impact. Mitigations describe the planned design or operational response; owners and statuses are tracked as the program progresses. All risks below align with the canonical data model: exact bytes everywhere, a single OS-agnostic storage collection schema, one envelope file per collection run containing a `hosts[]` array, weak linkage by keys (`backing_ref` / `group_ref`), and downstream derivation of `storage_class`, percentages, growth rates, deviations, and severities.

## Top risks and response

The following narrative covers the Critical and High risks that most threaten the objectives and require the most deliberate response.

- **RSK-04 (False positives / negatives): Critical.** Detection errors directly defeat objectives O1 (deviation detection) and O5 (risk reduction / compliance) and destroy operator trust. Response: per-OS / fstype / role threshold calibration, severity tiers, automatic flagging of high-impact deviations (withheld from auto-confirmation and surfaced distinctly via confidence-based triage), and continuous precision/recall tracking against a labeled set before any auto-action.

- **RSK-07 (Single-run, no history): Critical.** Without a time series there is no growth trend, so objectives O3 (reliability / full-filesystem prevention) and O4 (operational efficiency / forecasting) cannot be met. Response: persist an append-only series of collection runs from day one, require a minimum number of points before forecasting, and gate growth-dependent objectives on history availability.

- **RSK-19 (btrfs size accounting on SUSE / SLES): Critical.** Plain `df` on btrfs with subvolumes and snapshots reports wrong bytes, corrupting every downstream used ratio, forecast, and deviation. Response: an fstype-aware collector that uses `btrfs filesystem usage` / qgroups (never plain `df`), validated against allocated extents, with the result normalized to exact bytes in the schema.

- **RSK-23 (Simultaneous multi-OS delivery): Critical.** Delivering AIX, all Linux distributions, and Windows as first-class in a single release multiplies scope and integration risk. Response: anchor everything on the single OS-agnostic schema with pluggable per-OS collectors and standards, run parallel workstreams against common contracts, and define an explicit MVP per objective on every OS.

- **RSK-17 / RSK-18 / RSK-22 (Windows first-class set): High.** Windows joins now at full depth: schema mapping (no LVM; Storage Spaces pools and virtual disks map to groups and volumes via the `*_kind` discriminators, while basic disks yield partitions-as-volumes with empty groups), agentless WinRM security, and a distinct ACL-based compliance model. Response: explicit Windows mapping rules with conformance tests on real runs, WinRM over HTTPS / Kerberos with vaulted least-privilege accounts, and a separate compliance evaluator keyed on `access_model` so NTFS ACL intent is never judged by POSIX mount-option rules.

- **RSK-02 / RSK-21 (Standards coverage and explosion): High.** Missing reference images (notably AIX) mean there is nothing to deviate against, while the OS x distribution x version x role matrix risks a combinatorial standards blow-up. Response: build the reference standards early and tame the combinatorics with layered inheritance, storing only deltas from OS-family defaults down to role and version. The reference standards / golden images are an explicit upcoming deliverable outside this document set.

- **RSK-05 / RSK-16 (LLM hallucination and over-automation): High.** Agentic LangGraph reasoning can invent facts or recommend actions without sufficient grounding. Response: ground every assertion in the collected data with deterministic checks gating the LLM narrative, operate in recommend-only mode in which the system never executes remediation, and keep an immutable audit trail. The deterministic rules perform all measurement and detection; the LLM layer only explains, prioritizes, and recommends. People read the reports and act on them outside the system; there is no human step inside the pipeline.
