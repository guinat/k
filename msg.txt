Objet : Collecte de métadonnées stockage (multi-OS) — quoi récupérer et
        sous quel format

Bonjour, 
Si j'ai bien compris ce projet, les objectifs sont :
- O1 détection d'écarts aux standards — repérer les serveurs dont le partitionnement / système de fichiers s'écarte du modèle attendu pour leur OS et leur version
- O2 optimisation (inutile/obsolète/mal placé) — récupérer de l'espace : données obsolètes, volumes sur-provisionnés, données applicatives posées sur des partitions système
- O3 fiabilité & prévention des saturations — détecter en amont les systèmes de fichiers qui vont se remplir (en octets ET en inodes) pour éviter l'incident
- O4 efficacité opérationnelle & prévision — uniformiser les structures de stockage et anticiper la croissance / les besoins en capacité
- O5 réduction du risque & conformité — garantir que seules les données appropriées résident sur les partitions système, et vérifier les options de montage / ACL (référentiels CIS)

Dans le but de repondre a ces 5 objectifs, j'aurai besoin de plus de données sur les serveurs,

CE QUE J'ATTENDS EN SORTIE (dans la limite du possible)
1. UN SEUL fichier JSON par passage de collecte : un objet enveloppe { snapshot_id, generated_at, host_count,
   hosts: [ … ] } où CHAQUE élément de hosts[] est le document d'UN
   serveur (schéma en annexe A). Si le parc est très volumineux, le même
   contenu en JSONL (un objet-hôte par ligne) est accepté.
2. Toutes les tailles en OCTETS exacts jamais arrondi, ni Go, ni blocs,
   ni pourcentage. 
3. Pour chaque filesystem : total, utilisé ET disponible (les trois),
   + les inodes là où ça existe (Unix).
4. Horodatage en UTC (ISO-8601) + un snapshot_id par passage, et collecte
   RÉPÉTÉE dans le temps (on ajoute, on n'écrase pas) — c'est ce qui rend
   la prévision possible.
5. Préserver le lien filesystem ↔ topologie via les clés backing_ref /
   group_ref (c'est ce qui permet de dire « système » vs « données »).
6. Métadonnées NON SENSIBLES uniquement : pas de nom de fichier, pas de
   propriétaire, pas de contenu (sur Windows : un RÉSUMÉ d'ACL, pas les
   ACL nominatives).
7. Un champ NON APPLICABLE à un OS doit être renvoyé à `null` (vérifié
   N/A), jamais omis — pour distinguer « N/A » de « pas collecté ».

DÉMARCHE PROPOSÉE
D'abord un ÉCHANTILLON représentatif si possible sur plusieur type de servers

Subject: Storage metadata collection (multi-OS) — what to collect and in
         which format

Hi,

If I understand this project correctly, the objectives are:
- O1 deviation detection vs standards — spot servers whose partitioning / filesystem layout differs from the model expected for their OS and version
- O2 optimization (useless/obsolete/misplaced) — reclaim space: obsolete data, over-provisioned volumes, application data sitting on system partitions
- O3 reliability & full-filesystem prevention — detect ahead of time the filesystems about to fill up (in bytes AND inodes) to avoid the incident
- O4 operational efficiency & forecasting — standardize storage structures and anticipate growth / capacity needs
- O5 risk reduction & compliance — ensure only appropriate data resides on system partitions, and check mount options / ACLs (CIS baselines)

To meet these 5 objectives, I need more data about the servers.

WHAT I EXPECT AS OUTPUT (best effort)
1. ONE single JSON file per collection run: a wrapper object { snapshot_id, generated_at, host_count,
   hosts: [ … ] } where EACH element of hosts[] is ONE server's document
   (schema in appendix A). For very large fleets, the same content as JSONL
   (one host object per line) is acceptable.
2. All sizes in EXACT BYTES — never rounded, no GB, no blocks, no percent.
3. Per filesystem: total, used AND available (all three), + inodes where
   they exist (Unix).
4. UTC (ISO-8601) timestamp + a snapshot_id per run, and REPEATED
   collection over time (append, never overwrite) — this is what makes
   forecasting possible.
5. Preserve the filesystem ↔ topology link via the backing_ref /
   group_ref keys (this is what lets us tell "system" vs "data").
6. NON-SENSITIVE metadata only: no filenames, no owners, no content (on
   Windows: an ACL SUMMARY, not the named ACLs).
7. A field NOT APPLICABLE to an OS must be returned as `null` (verified
   N/A), never omitted — to distinguish "N/A" from "not collected".

PROPOSED APPROACH
First a representative SAMPLE, ideally across several server types.

      "size_free_bytes":  number,            // OBL — octets
      "member_disk_count":number,            // OBL

      "volume_name": "string",               // OBL (si volume présent)
      "volume_kind": "string",               // OBL — lvm-lv | partition | virtual-disk
      "group_ref":   "string|null",          // OBL — clé vers groupe parent
      "size_bytes":  number,                  // OBL — octets
      "type":        "string|null",          // REC — jfs2 | linear | striped | mirror | gpt-part ...
      "redundancy":  "string|null"           // COND — none | mirror | parity ...

    "disks": [{
      "disk_name":  "string",                // OBL (si disque exposé)
      "size_bytes": number,                   // OBL — octets
      "free_bytes": "number|null",           // REC — octets
      "group_ref":  "string|null",           // REC — clé vers groupe
      "media_type": "string|null"            // REC — hdd | ssd | unspecified
